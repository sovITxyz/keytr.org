keytr

Encrypt your nsec with a passkey. Log in to any Nostr client with your fingerprint.

The Problem

Nostr private keys (nsec) are hard to manage. You have to copy them between devices, paste them into clients, and hope nothing leaks. One mistake and your identity is compromised forever.

keytr fixes this. Your nsec gets encrypted with a passkey (fingerprint, Face ID, hardware key) and published to Nostr relays. On any new device, authenticate with your passkey to decrypt. No seed phrases. No clipboard. No extensions.

How It Works

  1. Generate or import your nsec in any keytr-compatible client
  2. Register a passkey — your device creates a credential bound to a gateway (e.g. keytr.org) or the client's own domain
  3. Your nsec is encrypted using a key derived from the passkey's PRF extension
  4. The encrypted blob is published as a kind:30079 event to your relays
  5. On a new device, open any keytr-compatible client and tap "Login with Passkey"
  6. Your passkey syncs via iCloud/Google/Windows — authenticate to decrypt your nsec

Encryption

Scheme

  • Key Source: WebAuthn PRF extension (hmac-secret)
  • KDF: HKDF-SHA256 with random 32-byte salt
  • Cipher: AES-256-GCM with 12-byte random IV
  • AAD: credential ID + version (prevents substitution)

The passkey never leaves your device. The PRF extension produces a deterministic secret that derives the encryption key. Even if the encrypted event is public, it is useless without your passkey.

Federated Cross-Client Login

WebAuthn binds passkey outputs to the domain (rpId) they were created on. To enable cross-client login, keytr uses a federated gateway model.

Any domain can become a passkey gateway by hosting a /.well-known/webauthn file that lists authorized client origins. Multiple gateways can coexist — no single domain controls the protocol.

Gateway Federation

  • keytr.org — primary gateway on Cloudflare (sovIT)
  • nostkey.org — alternative gateway on GitHub Pages (sovIT)
  • Any other domain — anyone can run their own gateway
  • Standalone — clients can use their own domain as rpId

Decentralize your keys: Register passkeys against multiple gateways, producing a separate kind:30079 event for each. keytr.org runs on Cloudflare, nostkey.org runs on GitHub Pages — different providers, different infrastructure, zero single points of failure. Lose access to one gateway? Your other events still work.

// Client uses a gateway as rpId
navigator.credentials.create({
  publicKey: {
    rp: { id: "keytr.org", name: "keytr" },
    extensions: {
      prf: { eval: { first: salt } }
    }
  }
})

// Or uses its own domain (standalone)
navigator.credentials.create({
  publicKey: {
    rp: { id: "myclient.com", name: "My Client" },
    extensions: {
      prf: { eval: { first: salt } }
    }
  }
})

Run Your Own Gateway

Decentralize the protocol further by running your own passkey gateway. All you need is a domain with one file:

# /.well-known/webauthn
{
  "origins": [
    "https://your-gateway.example",
    "https://client-a.com",
    "https://client-b.com"
  ]
}

Clients listed in your origins can register passkeys under your domain's rpId. The browser verifies authorization automatically via the Related Origin Requests spec.

The Event (NIP-79)

Encrypted keys are stored as kind:30079 parameterized replaceable events:

{
  "kind": 30079,
  "content": "<base64 encrypted nsec blob>",
  "tags": [
    ["d", "<credential-id>"],
    ["rp", "keytr.org"],
    ["algo", "aes-256-gcm"],
    ["kdf", "hkdf-sha256"],
    ["v", "1"]
  ]
}

Multiple passkeys can be registered — each produces a separate event with a different d tag. Lose one passkey, your others still work.

For Client Developers

Integrate keytr into your Nostr client with a few lines:

npm install keytr
import { registerPasskey, encryptNsec, decryptNsec } from 'keytr'

// Register
const cred = await registerPasskey({
  rpId: 'keytr.org',
  rpName: 'keytr',
  userName: npub
})

// Encrypt & publish
const blob = await encryptNsec({ nsec, credential: cred })

// Decrypt on new device
const { nsec } = await decryptNsec({
  encryptedBlob: blob,
  credentialId: cred.credentialId,
  rpId: 'keytr.org'
})

To have your origin authorized for cross-client login, add your domain to the origins list via PR.
